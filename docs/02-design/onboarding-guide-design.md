## Goal

Give a first-time Stellavault Desktop user a complete mental model of the app in under a minute, then get out of the way. Today there is no first-run experience at all: a new user lands on `DailyBrief`, sees a spinner, then vault stats, with zero explanation of what the app *is* or how to use its core surfaces (wikilinks, capture, the three discovery surfaces, the 3D graph, AI). The "code is there, but a new user can't find it" gap is the problem.

Concretely, onboarding must:

1. **Teach what-it-is up front** — a local, zero-API-key knowledge base — with one explicit, unmissable value prop.
2. **Hand off a usable map**: capture & write → find & connect (with live, OS-correct shortcuts) → AI, in a sequence a first-timer can finish in ~30s.
3. **Be trivially dismissible** and **never nag** — skippable at every step, shown exactly once, re-openable forever.
4. **Reinforce in-context** via richer empty-states and a small per-area micro-hint layer so guidance also appears exactly where a stuck user looks.
5. Ship **bilingual (ko + en)** and be **buildable by a solo dev** with no new dependencies and no fragile DOM-geometry code in the MVP.

**Non-goal for the MVP:** spotlight/coachmark positioning against live chrome (`getBoundingClientRect` cutouts). Per all four judges this is the most expensive-to-harden code a solo dev can own, and it is deferred to a clearly-bounded Phase 3 extension.

## New-user flow (step by step)

The spine is a skippable, dot-paginated **Welcome modal** rendered inside the existing `ui/Modal.tsx` primitive (Esc + backdrop-click + X close already built in, CSS-var themed, `width` prop). Four steps:

1. **First launch** — App boots; `TitleBar` (with `VaultSwitcher`), an empty `Sidebar`, and `EditorArea`→`DailyBrief` render exactly as today. Settings hydrate from the main process. Because `settings.onboarding?.completedVersion` is absent/older than `ONBOARDING_VERSION`, a one-shot effect (gated on `settingsHydrated`, mirroring the existing `App.tsx:117-124` pattern) calls `setWelcomeOpen(true)`. This fires the moment settings hydrate — it does **not** wait on `coreReady`, so a slow knowledge-engine boot never delays or races the modal.

2. **Step 1 — Welcome / what-it-is.** Greets the user and states the value prop plainly: *"Stellavault is a local, zero-API-key knowledge base — your notes never leave your machine."* Shows a live vault stat read from the same `core:get-stats` the `DailyBrief` already calls: `"Indexing 1,200 notes…"` while core boots, then `"1,200 notes ready"` (or, when `documentCount === 0`, `"Your vault is empty — let's create your first note."`). **Skip** sits bottom-left on every step.

3. **Step 2 — Capture & write.** Explains that `[[wikilinks]]` auto-create the target note on click (the real behavior in `WikilinkNode.ts`), and that dropping any file or link **anywhere in the window** captures it into the vault (drop-to-capture already exists; reuses the existing `capture.hint` string).

4. **Step 3 — Find & connect.** Shows the four discovery surfaces, each with its **real keybinding** resolved live from the command registry via `bindingFor(getCommand(id), settings.hotkeys)` and rendered through `formatChord()` (correct ⌘/Ctrl glyph per-OS, reflects any user rebind): Command palette (`mod+shift+p`), Quick switcher (`mod+p`), Search (`mod+shift+f`), 3D Graph (`mod+g`). One line each. This is the muscle-memory payload — the surfaces are named with their actual triggers, not screenshots.

5. **Step 4 — AI & finish.** Notes that AI (Ask / Synthesis / Coach) is **optional and works with no API key** via the extractive fallback, and that first use downloads a small (~30 MB) local model. Footer: a **primary** `"Create your first note"` button (runs the existing `file.new-note` command), and a **secondary** `"Set up AI"` button deep-linking Settings → AI tab via `setSettingsOpen(true, 'ai')`.

6. **Close / persist.** Any close path — Start/Create, Skip, Esc, X, backdrop click — writes `settings.onboarding = { completedVersion: ONBOARDING_VERSION }` **once** via `settingsStore.update()`. (One write, not many; this sidesteps the renderer's shallow optimistic-merge behavior in `settings-store.ts` that nested concurrent writes would expose.)

7. **Landing.** Clicking "Create your first note" closes the modal and drops the user into a fresh editor showing the enriched placeholder. If the vault was empty and they instead dismissed, the `Sidebar` file tree shows the upgraded "create your first note" empty-state and `DailyBrief` shows the dismissible getting-started strip.

8. **Re-open, anytime.** A registered command `help.welcome` ("Welcome / Getting started", category `Help`) makes the tour reachable forever from the command palette, the Help app-menu (next to `help.about` / `help.github`), and any hotkey the user binds in Settings → Hotkeys. The modal never auto-shows again unless `ONBOARDING_VERSION` is bumped in a future release.

9. **Returning user.** `completedVersion === ONBOARDING_VERSION` → the effect no-ops, the modal never mounts, nothing changes for existing users.

All strings via `useT()`.

- `packages/desktop/src/renderer/components/sidebar/FileTree.tsx` (around line 332) — replace bare `"No .md files found"` with a teaching empty-state: a short line + a **`[Create note]` button** that calls the existing new-note path, plus the live new-note shortcut and the drop affordance.
- `packages/desktop/src/renderer/components/editor/MarkdownEditor.tsx` (line 120) — make the **already-present** `Placeholder` i18n-driven (`t('editor.placeholder')`) and enrich its copy to mention `[[` and `/`. - `packages/desktop/src/renderer/components/shared/DailyBrief.tsx` — add a dismissible **getting-started strip** at the top of `briefWrapper`, rendered only when `stats.documentCount === 0 && onboarding.completedVersion is set/first-run && !hints['welcome']`: three shortcut chips (Command palette / New note / Open graph) each showing its live shortcut, plus a "Tour" link that runs `help.welcome` and a ✕ that dismisses it forever (`hints['welcome'] = true`). **Auto-dismiss condition** (closing Approach 1's noted gap): the strip is suppressed once `documentCount > 0`, so it never lingers for users who already have notes.

## Settings flag & persistence

- **Where:** in `AppSettings`, **not** `localStorage`. This matches every other persisted UI choice, round-trips through the existing `settings:set` IPC → `validateSettingsPatch` → deep-merge `set()` → atomic write → `settings:changed` broadcast, survives reinstall/cache-clear, and is consistent across windows.
- **Shape:** `onboarding?: { completedVersion?: number; hints?: Record<string, boolean> }`.
  - `completedVersion` — the modal's "seen it" gate. First-run = `completedVersion === undefined` after `settingsHydrated`. Re-show on a future major release by bumping `ONBOARDING_VERSION` (used sparingly to avoid nagging existing users on every release).
  - `hints` — Phase-2 per-area micro-hint seen-map; `{}` by default.
- **Write discipline:** the modal writes its flag **exactly once** at close. Phase-2 hints each write their own key on dismiss. The deep-merge in main never clobbers sibling keys; the renderer's optimistic state merge is shallow, so the design deliberately avoids rapid concurrent nested writes to the same slice (a known sharp edge the judges flagged in the hints-centric approach).
- **First-run detection is `coreReady`-independent:** gated only on `settingsHydrated`, so it fires deterministically with no timer/race regardless of how long core init takes.
- **Validation:** the new `settings-validate.ts` guard sanitizes the slice (numeric `completedVersion`, boolean-valued `hints` object) before it persists, per the project's whitelist convention.

## Empty-state guides

The non-modal half — guidance exactly where a stuck new user looks, with **minimal ongoing chrome** (no persistent banners). The empty state *is* the tutorial wherever cheap:

- **File tree** (`FileTree.tsx:~332`): replace `"No .md files found"` with `empty.fileTree.title` + `empty.fileTree.body` (live new-note shortcut + drop affordance) **and a `[Create note]` button** wired to the existing new-note command. A blank sidebar becomes a first action. (Grafted from Approach 3's do-by-doing empty-states.)
- **Editor** (`MarkdownEditor.tsx:120`): the existing TipTap `Placeholder` becomes i18n-driven (`editor.placeholder`) with enriched copy teaching `[[` and `/`. Shown only when the doc is empty (TipTap default). Zero new dep.
- **DailyBrief getting-started strip**: three live-shortcut chips (palette / new note / graph) + "Tour" link + ✕, shown only on first-run **empty** vaults, auto-suppressed once `documentCount > 0`. This is the only first-run residue on the home view and it self-clears.
- **Copy-only upgrades** (Phase 2, low-risk text swaps): Search panel empty result → operators guide (`tag:` / `path:` + cross-vault tip); Tags panel → "tags appear after indexing" (not just "No tags yet"); Backlinks/Outline/Related → "appear after indexing / try `@mention` or heading patterns." These swap hardcoded English for `empty.*` keys and add no layout.

## Micro-hints system

A **Phase-2** layer (explicitly *not* MVP), grafted from Approach 3 but built dependency-light and without the spotlight geometry:

- **Trigger:** a single `HintConductor` mounted once in `App.tsx` subscribes to the real panel choke-point `app-store.rightPanel` (and `coreReady`). The **first** time a panel opens — and only if its hint is unseen and not core-gated-out — it surfaces that area's hint. Editor-focus and drag-capture hints fire from their own components.
- **Presentation (MVP-of-the-layer):** to avoid the fragile `getBoundingClientRect` spotlight path the judges penalized, the first iteration presents each hint as a **non-anchored dismissible card** (toast-style, bottom-right, reusing `lib/toast.ts` styling/`theme.css` vars) rather than a positioned coachmark. One hint visible at a time (module-level single-active guard). Each has a "Got it" (`common.gotIt`) action that persists `hints[id] = true`. This delivers in-context teaching with no DOM-measurement risk.
- **Content:** `search` (operators + semantic), `graph` ("drag to rotate, click a node to preview; first open loads the 3D engine"), `ai` ("first open downloads a ~30 MB local model, then it's instant and offline"), `editor` (`[[` and `/`), `capture` (drop-anywhere). Each fires once, only when its area is reached.
- **Reset:** "Reset onboarding hints" button in Settings → General clears `onboarding.hints` (behind `ConfirmModal`) so demos/screenshots/support can replay everything.
- **Optional Phase-3 upgrade:** swap the toast-card for true anchored coachmarks (`data-coach` attributes + `getBoundingClientRect` + resize/`ResizeObserver`), pointing at the exact `TitleBar` icons — only after the cheaper card version proves the trigger logic.

## Phased rollout (MVP first, then extensions)

**Phase 1 — MVP (S–M, ~1 day).** The Welcome modal end-to-end + the three empty-states. Specifically: `WelcomeModal.tsx` + `onboarding-const.ts`; `welcomeOpen`/`setWelcomeOpen` + `help.welcome` command + `SettingsTabId` widening in `commands.ts`; `App.tsx` mount + one-shot first-run effect; the `onboarding` field in `ipc-types.ts` / both stores; the `settings-validate.ts` guard; ~24 bilingual i18n keys; `FileTree` create-note empty-state; `MarkdownEditor` placeholder i18n; `DailyBrief` getting-started strip (with auto-suppress); AppMenu Help entry. This ships the full "GET it fast" value with **no fragile DOM geometry** and a single optional settings field.

**Phase 2 — Micro-hint layer (M, ~1 day).** `lib/onboarding.ts` registry + `useHint`; `HintConductor` toast-card hints on first panel open; the `hints` half of the settings slice (already typed/guarded in Phase 1); ~10 hint i18n keys; copy-only empty-state upgrades on Search/Tags/Backlinks/Outline; "Reset onboarding hints" in Settings → General.

**Phase 3 — Anchored coachmarks (optional, M).** Promote the toast-cards to anchored spotlight coachmarks (`data-coach` + geometry + `ResizeObserver`) only if Phase 2 validates demand and the trigger logic. This is where the deferred fragile work lives, isolated to one component, behind everything that already works without it.

**Phase 4 — Versioned re-onboarding (XS).** Wire a "What's new" mode: on a major UI overhaul, bump `ONBOARDING_VERSION` to re-show a trimmed modal. Used sparingly.

## Risks & non-goals

**Risks**

- **Upfront modal friction.** A 4-step modal is more interruption than pure empty-states. *Mitigation:* a prominent always-visible Skip + close-on-anything; static, skimmable, ~30s total; shown once.
- **Static content can drift.** Shortcut copy assumes registry command ids stay stable. *Mitigation:* shortcuts are pulled **live** via `bindingFor`/`formatChord` (not hardcoded), and the vault stat reuses `core:get-stats` — nothing copy-pasted to go stale. Only command **ids** must stay stable (they're central; low risk).
- **`SettingsTabId` break.** `setSettingsOpen(true,'ai')` needs the union widened or `tsc` fails. *Mitigation:* called out as a required edit in Phase 1.
- **Nested-write clobber.** The renderer's shallow optimistic merge could transiently re-surface a just-dismissed hint under rapid writes. *Mitigation:* the modal writes once; Phase-2 hints write distinct keys with a single-active guard; the persisted deep-merge is per-key safe.
- **Poisoned settings patch.** A new persisted, renderer-written field is an injection surface. *Mitigation:* the `settings-validate.ts` guard (Phase 1) sanitizes the slice before persistence, per the Threat Model Gate.
- **Getting-started strip lingering** (Approach 1's noted gap). *Mitigation:* auto-suppressed once `documentCount > 0`.

**Non-goals**

- **Live-DOM spotlight/coachmark positioning in the MVP** — deferred to Phase 3; the cheaper toast-card hints cover Phase 2.
- **Branching/adaptive/interactive tutorials** — the modal teaches surfaces, not hands-on practice; the empty-states and hints supply the "do-by-doing" first actions.
- **Forcing the tour** — never blocking beyond the first dismissible modal; no auto-replay except an explicit `ONBOARDING_VERSION` bump.
- **`localStorage` for the flag** — rejected in favor of `AppSettings` (consistency, survives reinstall, cross-window).
- **A new dependency for the editor placeholder** — `@tiptap/extension-placeholder` is already bundled and configured.
- **Telemetry / first-run analytics** — out of scope (project default is telemetry OFF).
