# Evan Knowledge Hub - Feature Discovery Analysis

> **분석일**: 2026-03-30 | **분석자**: PM Lead Agent
>
> 확정된 핵심 기능(Phase 1-3)을 건드리지 않고, 포지셔닝("AI의 뇌, 장기 기억")을 강화하는 추가 기능을 발굴합니다.
> 경쟁사 조사 및 시장 트렌드 기반으로 **20개 기능**을 제안합니다.

## 경쟁사 갭 분석 요약

| 경쟁사 | 그들이 하는 것 | 그들이 하지 않는 것 (= 우리의 기회) |
|--------|--------------|-----------------------------------|
| **Khoj** | 다중 소스 RAG, self-host, 에이전트 자동화 | 3D 시각화, 지식 변화 추적, 기억 감쇠 모델, 개발자 워크플로우 통합 |
| **Smart Connections** | Obsidian 네이티브 임베딩, 사이드바 관련 노트 | MCP 통합, 외부 AI 도구 연결, 시각화, 지식 인텔리전스 |
| **NotebookLM** | Google AI 기반 요약/대화, 비디오 개요 생성 | 로컬-퍼스트, PKM 직접 통합, 3D 시각화, 개발자 도구 연동 |
| **Mem.ai** | 자동 정리/태깅, 깔끔한 UX | 오픈소스, 로컬 데이터 주권, MCP, 시각화, 커스터마이징 |
| **Think Machine** | 3D 마인드맵, Wayfinder AI | PKM 도구 직접 통합, MCP, 로컬-퍼스트, 개발자 워크플로우 |
| **Vestige** (신흥) | FSRS-6 기억 감쇠, 3D 뉴럴 대시보드, MCP | 개인 지식 통합(노트/문서), 다중 소스, 지식 그래프 시각화 |
| **InfraNodus** | 텍스트 네트워크 분석, 갭 탐지, 3D 그래프 | MCP 통합, 로컬-퍼스트, 개발자 워크플로우, 다중 소스 |

**핵심 발견**: Vestige가 "AI 에이전트의 기억"이라는 비슷한 포지션을 잡고 있지만, 개인 지식(노트/문서)이 아닌 에이전트 세션 메모리에 집중함. InfraNodus가 지식 갭 탐지를 하지만 클라우드 전용. 이 두 영역의 교집합이 우리의 블루오션.

## Feature Proposals (20개)

### Category 1: 지식 인텔리전스 (단순 검색 -> 인사이트 추출)

---

#### F01. Knowledge Gap Detector (지식 공백 탐지기)

**설명**: 지식 그래프에서 연결이 약하거나 없는 영역을 AI가 자동 탐지하여 "이 주제에 대한 노트가 부족합니다"를 알려줌

**사용자 Pain**: 3000개 노트가 있어도 자기가 모르는 것이 뭔지 모름. "알고 있는 것"은 보이지만 "알아야 하는데 모르는 것"은 보이지 않음.

| 항목 | 내용 |
|------|------|
| 구현 난이도 | Medium |
| 차별화 가치 | **높음** - InfraNodus가 유사 기능을 하지만 클라우드 전용이고 PKM 통합 없음. 대기업은 "개인 지식의 빈틈"에 관심 없음 |
| 추천 Phase | Phase 2 (3D 그래프와 함께, 빈 영역을 시각적으로 강조) |
| 구현 방향 | 그래프 클러스터 간 브릿지 노드 부족 탐지 + 토픽 모델링으로 커버리지 분석 |

---

#### F02. Knowledge Evolution Timeline (지식 진화 타임라인)

**설명**: 특정 주제에 대한 자신의 이해가 시간에 따라 어떻게 변화했는지 추적/시각화

**사용자 Pain**: "내가 React에 대해 1년 전에 쓴 것과 지금 아는 것이 얼마나 달라졌지?" -- 자기 성장을 측정할 수 없음

| 항목 | 내용 |
|------|------|
| 구현 난이도 | Medium |
| 차별화 가치 | **매우 높음** - 경쟁사 중 아무도 하지 않음. 대기업도 "개인 지식의 시간적 변화"에 관심 없음 |
| 추천 Phase | Phase 2 (3D 그래프에 시간 축 추가 -- 4D 시각화) |
| 구현 방향 | 노트 메타데이터(created/modified) + 임베딩 변화 추적. 같은 주제 노트의 시맨틱 드리프트 계산 |

---

#### F03. Contradiction Detector (모순 탐지기)

**설명**: 서로 다른 노트에서 상충되는 정보를 AI가 발견하여 알려줌

**사용자 Pain**: 2년 전에 쓴 노트와 최근 노트가 상반된 정보를 담고 있어도 모름. "어제의 나"와 "오늘의 나"가 다른 걸 말하고 있을 때.

| 항목 | 내용 |
|------|------|
| 구현 난이도 | High |
| 차별화 가치 | **매우 높음** - 경쟁사 중 아무도 하지 않음. 학술 연구 수준의 도전적 기능 |
| 추천 Phase | Phase 3+ (Post-MVP, AI 모델 필요) |
| 구현 방향 | 동일 토픽 노트 클러스터 내에서 NLI(Natural Language Inference) 모델로 entailment/contradiction 판별 |

---

#### F04. Auto Knowledge Digest (자동 지식 요약 뉴스레터)

**설명**: 주간/월간으로 "이번 주에 추가된 지식", "가장 활발하게 변화한 영역", "오래된 미확인 노트" 등을 정리하여 로컬 HTML 리포트 생성

**사용자 Pain**: 노트를 써놓고 안 돌아보는 문제. 지식이 쌓이기만 하고 활용되지 않음.

| 항목 | 내용 |
|------|------|
| 구현 난이도 | Low-Medium |
| 차별화 가치 | **중간** - Khoj가 뉴스레터 자동화를 하지만, 개인 지식 변화 추적은 아님 |
| 추천 Phase | Phase 2 |
| 구현 방향 | 벡터 DB 메타데이터 분석 + LLM 요약. Cron job 또는 CLI 명령 |

---

#### F05. Semantic Clustering & Auto-Tagging (시맨틱 클러스터링 + 자동 태깅)

**설명**: 임베딩 기반으로 노트를 자동 클러스터링하고, 각 클러스터에 AI가 태그/레이블 부여

**사용자 Pain**: Obsidian 헤비 유저의 고질적 문제 -- "태깅 시스템이 무너졌다". 수동 태깅은 지속 불가능.

| 항목 | 내용 |
|------|------|
| 구현 난이도 | Medium |
| 차별화 가치 | **중간-높음** - Mem.ai가 자동 정리를 하지만 클라우드 전용. 로컬에서 이걸 하는 도구는 없음 |
| 추천 Phase | Phase 1.5 (벡터화 직후, 3D 그래프 전) |
| 구현 방향 | HDBSCAN 클러스터링 + LLM 레이블링. 결과를 Obsidian frontmatter에 역주입 가능 |

---

### Category 2: 시각화 확장 (3D 그래프 외 새로운 방식)

---

#### F06. Knowledge Heatmap (지식 히트맵)

**설명**: 3D 그래프 위에 활동 빈도/최신성을 히트맵으로 오버레이. "뜨거운 영역" = 최근 활발, "차가운 영역" = 방치된 지식

**사용자 Pain**: 3D 그래프가 "구조"를 보여줘도 "활력"을 보여주지 않음. 어떤 영역이 활발하고 어떤 영역이 죽었는지 한눈에 파악 불가.

| 항목 | 내용 |
|------|------|
| 구현 난이도 | Low |
| 차별화 가치 | **높음** - Think Machine도 InfraNodus도 시간 기반 활동 히트맵은 없음 |
| 추천 Phase | Phase 2 (3D 그래프 확장 기능) |
| 구현 방향 | 노드 색상/크기를 수정일/접근 빈도에 매핑. Three.js shader로 구현 |

---

#### F07. Knowledge Constellation View (별자리 뷰)

**설명**: 주제별 클러스터를 "별자리"로 표현. 각 별자리를 탭하면 해당 주제의 노트가 펼쳐짐. 줌아웃하면 전체 "지식 우주"

**사용자 Pain**: 3D 그래프가 노드 수백 개일 때 복잡해짐. 높은 추상화 수준의 뷰가 필요.

| 항목 | 내용 |
|------|------|
| 구현 난이도 | Medium |
| 차별화 가치 | **매우 높음** - 독창적인 UX. 마케팅/바이럴에 극도로 유리. "내 지식 우주" 스크린샷 공유 |
| 추천 Phase | Phase 2.5 (모션 제어와 함께 -- "손으로 별자리를 탐험하다") |
| 구현 방향 | HDBSCAN 클러스터 → Semantic LOD (Level of Detail). 줌 레벨에 따라 별자리 ↔ 개별 노트 전환 |

---

#### F08. Knowledge River Flow (지식 흐름도)

**설명**: 노트 간의 참조/인용 방향을 시각화. "이 아이디어가 어떤 노트에서 시작해서 어떻게 발전했는지"를 강 흐름처럼 표현

**사용자 Pain**: 링크만 있으면 "관계"는 보이지만 "흐름"(어떤 것이 어떤 것에서 유래했는지)은 보이지 않음.

| 항목 | 내용 |
|------|------|
| 구현 난이도 | High |
| 차별화 가치 | **높음** - 방향성 있는 지식 흐름 시각화는 어디에도 없음 |
| 추천 Phase | Phase 3+ |
| 구현 방향 | 노트 내 백링크/포워드링크 분석 + 생성일 기반 방향 추론. Sankey/Flow 다이어그램 렌더링 |

---

### Category 3: 기억 강화 (AI의 장기 기억 역할 강화)

---

#### F09. Knowledge Decay Model (지식 감쇠 모델)

**설명**: FSRS(Free Spaced Repetition Scheduler) 알고리즘을 적용하여, 오래 사용하지 않은 지식의 "기억 강도"를 추적하고, 중요한 것을 주기적으로 리마인드

**사용자 Pain**: 6개월 전에 공부한 디자인 패턴을 완전히 잊었는데, 노트는 있으나 존재 자체를 모름.

| 항목 | 내용 |
|------|------|
| 구현 난이도 | Medium |
| 차별화 가치 | **매우 높음** - Vestige가 AI 에이전트 메모리에 FSRS를 적용하지만, 개인 지식(노트)에 적용하는 것은 아무도 안 함 |
| 추천 Phase | Phase 2 (3D 그래프에서 감쇠 중인 노드를 시각적으로 "흐려지게" 표현) |
| 구현 방향 | FSRS-6 파라미터로 각 노트의 기억 강도 계산. 접근/활용 시 리셋. MCP 검색 시에도 decay weight 반영 |

---

#### F10. Context Snapshot (컨텍스트 스냅샷)

**설명**: 현재 작업 중인 프로젝트의 관련 지식을 "스냅샷"으로 묶어 저장. AI가 이 스냅샷을 로딩하면 즉시 해당 프로젝트 맥락에 진입

**사용자 Pain**: 프로젝트를 3개 병행하면 각 프로젝트의 "맥락"을 로딩하는 데 시간이 걸림. "아 이 프로젝트 어디까지 했더라?"

| 항목 | 내용 |
|------|------|
| 구현 난이도 | Low-Medium |
| 차별화 가치 | **높음** - 개발자의 컨텍스트 스위칭 비용(주당 5시간 손실)을 직접 해결 |
| 추천 Phase | Phase 1 (MCP 서버 기능 확장) |
| 구현 방향 | MCP tool: `create-snapshot`, `load-snapshot`. 관련 문서 ID 목록 + 요약을 JSON으로 저장 |

---

#### F11. Adaptive Memory Priority (적응형 기억 우선순위)

**설명**: MCP 검색 시, 사용자의 현재 컨텍스트(작업 중인 파일, 최근 쿼리)를 기반으로 검색 결과 우선순위를 동적 조정

**사용자 Pain**: 같은 "authentication" 검색어라도, React 프로젝트 작업 중이면 React 관련 노트가, Go 프로젝트 중이면 Go 관련 노트가 먼저 나와야 함.

| 항목 | 내용 |
|------|------|
| 구현 난이도 | Medium |
| 차별화 가치 | **높음** - MCP 서버에서 "현재 작업 컨텍스트 인지"는 아무도 안 함 |
| 추천 Phase | Phase 1 (MCP 서버 고급 기능) |
| 구현 방향 | MCP 쿼리 시 현재 파일 경로/언어/최근 쿼리 히스토리를 가중치로 RRF에 반영 |

---

#### F12. Cross-Vault Knowledge Federation (크로스 볼트 연합)

**설명**: 여러 Obsidian vault (개인, 업무, 프로젝트별)를 하나의 통합 검색/그래프로 연합하되, 물리적으로는 분리 유지

**사용자 Pain**: 개인 노트와 업무 노트를 분리하고 싶지만, 검색/연결은 통합하고 싶음. "개인 학습 노트에서 업무 관련 인사이트를 찾고 싶은데..."

| 항목 | 내용 |
|------|------|
| 구현 난이도 | Medium |
| 차별화 가치 | **높음** - Smart Connections는 단일 vault만 지원. Khoj도 vault 간 연합 검색은 없음 |
| 추천 Phase | Phase 3 (다중 소스 확장과 함께) |
| 구현 방향 | 각 vault 별도 벡터 DB → 쿼리 시 federation search. 3D 그래프에서 vault 별 색상 구분 |

---

### Category 4: 개발자 워크플로우 (Claude Code/Cursor 특화)

---

#### F13. CLAUDE.md Auto-Generator (CLAUDE.md 자동 생성기)

**설명**: Obsidian vault에서 프로젝트 관련 지식을 추출하여 Claude Code용 CLAUDE.md를 자동 생성/업데이트

**사용자 Pain**: CLAUDE.md를 수동으로 관리하는 것이 번거로움. 프로젝트 이력/패턴이 있는데 매번 직접 정리해야 함.

| 항목 | 내용 |
|------|------|
| 구현 난이도 | Low-Medium |
| 차별화 가치 | **매우 높음** - Claude Code 사용자의 핵심 Pain을 직접 해결. 대기업이 이걸 만들 이유 없음 (특정 도구 조합) |
| 추천 Phase | Phase 1 (MCP 서버 출시와 동시에 -- 킬러 유스케이스) |
| 구현 방향 | 프로젝트명 기반 관련 노트 검색 → 아키텍처/패턴/결정사항 요약 → CLAUDE.md 템플릿에 채움 |

---

#### F14. Decision Journal (결정 저널)

**설명**: 코딩 중 내린 기술적 결정(라이브러리 선택, 아키텍처 패턴 등)을 구조화하여 자동 기록. 나중에 "왜 이 선택을 했지?"에 답변

**사용자 Pain**: 3개월 전에 왜 Zustand 대신 Jotai를 선택했는지 기억 안 남. ADR(Architecture Decision Record)을 쓰자니 번거로움.

| 항목 | 내용 |
|------|------|
| 구현 난이도 | Low |
| 차별화 가치 | **높음** - ADR을 자동화하는 도구 없음. MCP를 통해 AI가 "이전에 비슷한 결정을 할 때..." 참조 가능 |
| 추천 Phase | Phase 1 (MCP tool: `log-decision`, `find-decisions`) |
| 구현 방향 | Obsidian에 `decisions/` 폴더 자동 생성. MCP tool로 구조화 입력. 검색 시 decision 타입 필터 |

---

#### F15. Code-to-Knowledge Linker (코드-지식 링커)

**설명**: 코드 파일/함수와 관련된 지식 노트를 자동으로 연결. "이 함수의 설계 배경이 어떤 노트에 있는지" 추적

**사용자 Pain**: 코드에 `// see design doc` 같은 주석을 달지만, 실제 문서 링크는 깨지거나 찾기 어려움.

| 항목 | 내용 |
|------|------|
| 구현 난이도 | Medium-High |
| 차별화 가치 | **매우 높음** - 코드와 개인 지식의 양방향 연결은 완전히 새로운 개념 |
| 추천 Phase | Phase 2 (3D 그래프에서 코드 파일 노드도 표시) |
| 구현 방향 | git 커밋 메시지/PR 설명에서 키워드 추출 → 관련 노트 매칭. MCP tool: `link-code-to-note` |

#### F16. Knowledge Graph Screenshot/Embed (그래프 스크린샷/임베드)

**설명**: 3D 지식 그래프의 특정 뷰를 이미지/인터랙티브 임베드로 내보내기. 블로그, README, 소셜미디어에 공유 가능

**사용자 Pain**: "내 지식 그래프 멋있다" → 공유하고 싶은데 방법이 없음. 스크린샷은 정적이고 인터랙션 불가.

| 항목 | 내용 |

| 추천 Phase | Phase 2 (3D 그래프와 동시 출시 -- 바이럴 엔진) |

**사용자 Pain**: 개발자 프로필에 "내가 실제로 아는 것"을 보여줄 방법 없음. GitHub 잔디밭은 "코딩량"이지 "지식"이 아님.

| 항목 | 내용 |

| 차별화 가치 | **높음 (바이럴)** - "GitHub 잔디밭의 지식 버전". 개발자 정체성에 호소. 대기업이 할 영역 아님 |
| 추천 Phase | Phase 2.5 |

**사용자 Pain**: Obsidian을 시작하면 "뭘 어떻게 정리하지?"가 최대 장벽. Zettelkasten 책을 읽을 시간 없음.

| 구현 방향 | JSON/YAML로 토픽 구조 정의. CLI로 vault에 scaffold 생성. GitHub repo로 커뮤니티 공유 |

## 결론 및 전략적 제언

### 핵심 메시지

1. **Phase 1의 킬러 유스케이스는 F13 (CLAUDE.md 자동 생성)**입니다. MCP + 개인 지식의 가치를 가장 빠르게 증명하며, Claude Code 사용자가 "5분 안에 왜 이걸 써야 하는지" 이해하게 합니다.

2. **Phase 2의 바이럴 엔진은 F16 (그래프 공유) + F07 (별자리 뷰)**입니다. 3D 그래프의 "와" 효과를 소셜미디어로 전파하는 것이 오픈소스 성장의 핵심입니다.

3. **Vestige와의 차별화는 "에이전트 메모리 vs 개인 지식"**입니다. Vestige가 "AI의 단기 기억"이라면, Evan Knowledge Hub는 "AI의 장기 기억 + 지식 체계"입니다. F09(감쇠 모델)로 이 포지션을 명확히 합니다.

4. **지식 인텔리전스 (F01, F02, F03)가 장기 해자(moat)**입니다. 검색은 모두 하지만, "모르는 것을 알려주고, 모순을 찾아주고, 변화를 추적하는" 도구는 없습니다.

### 1인 개발자 전략

20개 중 Phase 1에 추가할 5개(F13, F10, F14, F20, F11)는 모두 **Low-Medium 난이도**입니다. 기존 MCP 서버에 tool을 추가하는 수준이므로, 핵심 기능 개발 일정에 큰 영향을 주지 않으면서 차별화 가치를 크게 높일 수 있습니다.
