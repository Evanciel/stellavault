## 요약

| 관점 | 내용 |
|------|------|
| **문제** | Stellavault는 강력하지만 고립된 지식 시스템으로 운영됩니다. 각 사용자의 금고는 인지적 섬으로, 다른 사람이 보유한 관련 지식을 발견하거나, 집단적 지식 공백을 파악하거나, Wikipedia나 Stack Overflow를 가치 있게 만드는 네트워크 효과를 활용할 수 없습니다. 지식 관리 시장(2025년 기준 230억 달러)은 사용자에게 데이터 소유권 포기를 요구하는 중앙화 플랫폼(Notion, Confluence)이 지배하고 있습니다. 절대적인 데이터 주권을 보존하면서 집단 지성을 가능케 하는 솔루션은 아직 존재하지 않습니다. |
| **솔루션** | Federation Protocol은 각 Stellavault 인스턴스를 주권 노드로 취급하여 P2P 네트워크를 통해 임베딩 벡터만 공유합니다(원문 텍스트는 절대 공유하지 않음). 노드 간 의미 검색, 네트워크 수준 갭 탐지, 지식 라우팅을 수행하여 프라이버시를 훼손하지 않으면서 집단 지성을 실현합니다. P2P 연결에 Hyperswarm, 공유 임베딩에 차등 프라이버시 노이즈, Web of Trust 평판 시스템을 사용하며, 기존 MCP 도구 생태계와 통합하여 AI 에이전트가 연합 네트워크에 투명하게 접근할 수 있습니다. |
| **주요 기능 / UX** | `stellavault federate join`으로 네트워크 연결; 기존 `search` CLI를 통한 연합 의미 검색; 3D 그래프 내 네트워크 갭 히트맵; 지식 라우팅("노드 C가 X에 대해 알고 있음"); 신뢰 점수; 기여 인센티브(검색 크레딧); 4개의 새로운 MCP 도구(`federated-search`, `network-gaps`, `route-query`, `trust-score`)를 통한 AI 에이전트 연합. |
| **핵심 가치** | "당신의 지식은 당신의 것으로 남습니다. 네트워크의 지성은 모두의 것이 됩니다. Stellavault Federation은 개인 금고를 분산형 두뇌로 전환합니다 -- 아무도 당신의 노트를 읽지 않지만, 모두가 당신의 이해로부터 혜택을 받습니다." |

## Part I: 발견 분석

> 프레임워크: Teresa Torres의 Opportunity Solution Tree (5단계 발견 체인)

### 1. 브레인스토밍 -- 기회 공간

| # | 기회 | 설명 | 사용자 시그널 |
|---|------|------|--------------|
| O1 | **볼트 간 지식 발견** | 사용자가 다른 사람이 보유한 관련 지식을 수동 질문이나 외부 플랫폼 검색 없이 찾을 수 없음 | "누군가 이미 이걸 알아냈을 텐데" -- 개발자들의 흔한 감상 |
| O2 | **집단 지식 공백 탐지** | 개별 갭 탐지기로는 커뮤니티 전반의 체계적 지식 공백을 놓침 | 커뮤니티가 같은 문제를 반복적으로 재발견함 |
| O3 | **프라이버시 보장 지식 공유** | 기존 공유 방식은 원시 데이터 노출을 요구함(GitHub, 위키, 포럼) | 독자적 연구/노트를 공개적으로 공유하기를 꺼림 |
| O4 | **AI 에이전트 네트워크 접근** | AI 에이전트가 단일 사용자의 지식 기반에 제한됨 | Claude/GPT 응답이 개별 컨텍스트 윈도우에 제한됨 |
| O5 | **탈중앙화 지식 라우팅** | 중앙화된 디렉토리 없이 "누가 무엇을 아는지" 알 방법이 없음 | 전문가 찾기는 20억 달러 규모의 기업 문제 |
| O6 | **커뮤니티 지식 건강** | 집단 지식 커버리지, 감쇠, 성장에 대한 지표 부재 | 오픈소스 커뮤니티에 지식 가관측성이 부족 |
| O7 | **지식 마켓플레이스** | 전문 지식을 수익화할 프라이버시 보장 방법이 없음 | 컨설턴트는 지식 접근이 아닌 시간을 판매 |

### 3. 우선순위 실험

| # | 가정 | 실험 | 성공 지표 | 노력 |
|---|------|------|----------|------|
| E1 | A1 (프라이버시 신뢰) | 사용자 설문: "텍스트 복원이 불가능하다고 보장된다면 임베딩 벡터를 공유하겠습니까?" + 기술 데모 | Obsidian 파워 유저의 >70%가 "예" | 낮음 |

| E5 | A6 (네트워크 효과) | 합성 볼트 데이터를 사용하여 노드 수 대비 연합 검색 품질 시뮬레이션 | 10개 노드에서 검색 관련성 >60%, 50개 노드에서 >80% | 중간 |

### 5. 선정된 상위 3개 기회

1. **O1 + O3: 프라이버시 보장 연합 의미 검색** -- 핵심 기능. 아무도의 노트도 읽지 않으면서 볼트 전체를 검색합니다. "10배 더 나은" 순간입니다.
2. **O5: 지식 라우팅** -- "이 질문은 노드 C가 가장 잘 답할 수 있습니다"라는 기능은 수동적 참여자에게도 즉각적이고 실질적인 가치를 제공합니다.
3. **O2 + O6: 네트워크 갭 탐지** -- 개인이 아닌 *커뮤니티*가 모르는 것을 발견하는 것은 고유하게 강력하며 기존 도구에 동등한 것이 없습니다.

### 6. 가치 제안 (JTBD 6파트 프레임워크)

| 파트 | 내용 |
|------|------|
| **1. 고객** | Obsidian 볼트를 주요 지식 시스템으로 사용하는 기술 지식 근로자(개발자, 연구자, PKM 열정가) |
| **2. 해결해야 할 과업** | 지식 공백이나 질문에 직면했을 때, 신뢰할 수 있는 네트워크에 누군가 관련 이해를 갖고 있는지 발견하고 싶습니다. 나와 상대방의 프라이버시를 훼손하지 않으면서 더 빨리 학습할 수 있도록 |
| **3. 고통 (현재)** | 현재 해야 하는 것: (a) 공개 인터넷 검색(노이즈가 많고 일반적), (b) 동료에게 직접 질문(높은 마찰, 사회적 비용), (c) 포럼에 게시(느리고 공개적). 모든 옵션이 프라이버시나 품질을 희생함 |
| **4. 이득 (미래)** | Federation 적용 시: 신뢰할 수 있는 볼트 전반의 즉각적인 의미 검색, 가장 관련성 높은 노드로의 자동 지식 라우팅, 커뮤니티 전체 갭 가시성, 이 모든 것이 내 노트가 내 기기를 떠나지 않으면서 가능 |
| **5. 대안** | 공유 위키(Notion, Confluence)는 중앙화를 요구함. Fediverse(Mastodon, Nostr)는 지식 구조가 아닌 콘텐츠를 공유함. 학술 연합 KG 연구는 기업 전용. 개인 볼트 연합을 위한 것은 존재하지 않음 |
| **6. 차별화** | 다음을 결합한 유일한 솔루션: (a) 임베딩만 공유(설계에 의한 프라이버시), (b) 기존 PKM 도구 통합(Obsidian), (c) MCP를 통한 AI 에이전트 접근, (d) 탈중앙화 아키텍처(중앙 서버 없음), (e) 네트워크 전반의 FSRS 기반 지식 건강 추적 |

### 8. SWOT 분석

| | 유리 | 불리 |
|---|------|------|
| **내부** | **강점:** 완전한 로컬 지식 플랫폼 이미 구축; SQLite-vec 임베딩 공유 준비 완료; Plugin SDK로 확장 가능; MCP 도구가 AI 에이전트 통합 제공; MIT 라이선스가 채택 촉진 | **약점:** 단일 개발자; 코드베이스에 P2P 네트워킹 경험 없음; all-MiniLM-L6-v2는 384차원(프로덕션 모델보다 작음); 모바일 클라이언트 없음; 인증/신원 시스템 미구축 |

#### 11.2 상세 분석

**C1: Solid 프로젝트 (Tim Berners-Lee / Inrupt)**
- *모델:* 사용자가 자신이 제어하는 서버의 "Pod"에 데이터를 저장. 앱이 WebID 인증을 통해 접근 요청.
- *강점:* W3C 표준(RDF, SPARQL), Tim Berners-Lee 지원, 기업 파트너십(BBC, NHS).
- *약점:* Pod를 위한 서버 인프라 필요, RDF 복잡성이 메인스트림 개발자를 소외, 의미 검색 기능 없음, 임베딩/AI 통합 없음.
- *연합 갭:* 임베딩이 아닌 실제 데이터를 공유함. 프라이버시가 아키텍처적 프라이버시가 아닌 접근 제어에 의존. 지식 특화 기능 없음.

**C2: AT Protocol (Bluesky)**
- *모델:* Personal Data Server(PDS)가 사용자 데이터 저장, Relay가 집계, AppView가 렌더링. IETF에서 표준화 중(2026년 1월).
- *강점:* 대규모 검증(4천만+ 사용자), 이동 가능한 신원, 알고리즘 선택(4만+ 피드), 활발한 개발.
- *약점:* 지식이 아닌 소셜 콘텐츠용으로 설계. 릴레이 아키텍처에 상당한 인프라 필요. 의미 검색 없음. 전체 콘텐츠 공유.
- *연합 갭:* 지식 중심이 아닌 콘텐츠 중심. 임베딩, 갭 탐지, FSRS 없음. 신원 모델(DID)은 재사용 가능.

**C3: Nostr 프로토콜**
- *모델:* 단순 릴레이 아키텍처 -- 클라이언트가 서명된 이벤트를 발행하고, 릴레이가 저장 및 전달. 최소한의 릴레이 로직.
- *강점:* 극도의 단순성, 검열 저항성, 1천만 달러+ 펀딩(Jack Dorsey), 성장하는 생태계.
- *약점:* 의미 이해 없음, 릴레이 스팸 문제, 지식 구조 없음, 검색 품질 보장 없음.
- *연합 갭:* "노트와 기타 것들" -- 지식 표현이 아닌 원시 콘텐츠를 전송. 전송 레이어로 사용 가능하지만 나머지 모든 것을 위에 구축해야 함.

**C4: Anytype**
- *모델:* IPFS 기반 암호화 동기화를 갖춘 로컬 우선. 오브젝트 기반(마크다운 아님). 비공개 소스.
- *강점:* 매끄러운 UX, 강력한 암호화, IPFS 백본, 성장하는 사용자 기반.
- *약점:* 폐쇄 생태계, 플러그인 시스템 없음, AI/MCP 통합 없음, 동기화는 연합이 아닌 복제, 사용자 간 검색 없음.
- *연합 갭:* 사용자 간이 아닌 기기 간 데이터 동기화. 임베딩 공유 없음, 집단 지성 기능 없음.

**C5: Logseq**
- *모델:* 로컬 마크다운/org-mode 파일. Git 또는 수동 내보내기로 공유. 오픈소스.
- *강점:* 오픈소스, 강력한 아웃라이너, 활발한 커뮤니티, Logseq Sync(유료).
- *약점:* 벡터 검색 없음, 임베딩 지원 없음, 연합 프로토콜 없음, 제한적 AI 통합, 동기화는 사용자 연합이 아닌 기기 동기화.
- *연합 갭:* 네트워크 스토리가 없는 순수 로컬 도구. 모든 것을 처음부터 구축해야 함.

### 20. P2P 프로토콜: Hyperswarm

#### 20.1 왜 Hyperswarm인가 (libp2p나 WebRTC가 아닌 이유)

| 기준 | Hyperswarm | libp2p | WebRTC Direct |
|------|:----------:|:------:|:-------------:|
| NAT 홀 펀칭 | 내장, 검증됨 | 나중에 추가, 덜 안정적 | STUN/TURN 필요 |
| 데스크톱 네이티브 | 예 (Node.js 우선) | 예 (다중 언어) | 브라우저 우선 |
| 복잡도 | 낮음 (~500 LOC 통합) | 높음 (모듈형, 많은 의존성) | 중간 |
| DHT 피어 발견 | 내장 Kademlia | 내장이나 더 무거움 | 없음 (시그널링 필요) |
| 연결 속도 | 일반적으로 <5초 | 일반적으로 5-15초 | 2-10초 + STUN 오버헤드 |
| 스트림 다중화 | Hypercore Protocol 경유 | yamux/mplex 경유 | 데이터 채널 경유 |
| 생태계 적합성 | Hypercore (append-only 로그에 적합) | IPFS (콘텐츠 주소 지정) | 없음 |
| 성숙도 | 7년+, Dat/Hypercore 생태계 | 10년+, IPFS 생태계 | 15년+, 보편적 |

**결정:** Hyperswarm이 단순성, NAT 처리, 데스크톱 네이티브 적합성에서 승리합니다. Stellavault는 Node.js CLI/데스크톱 도구이며 -- Hyperswarm은 이를 위해 만들어졌습니다. libp2p의 브라우저 우선 최적화와 모듈형 복잡성은 불필요한 오버헤드입니다.

**대안:** Hyperswarm이 유지보수되지 않게 되면, 전송 레이어가 `FederationTransport` 인터페이스 뒤에 추상화되어 있어 libp2p로 마이그레이션이 가능합니다.

#### 20.2 네트워크 토폴로지

```
                 [부트스트랩 DHT 노드]
                   (3-5개 상시 가동)
                  /       |        \
                 /        |         \
           [Node A]---[Node B]---[Node C]
               \         |         /
                \        |        /
                 [Node D]---[Node E]
                       |
                    [Node F]
                  (프리미엄 릴레이)
```

- **부트스트랩 노드:** 초기 피어 발견을 위한 최소한의 상시 가동 DHT 노드. 데이터를 저장하지 않음. 커뮤니티 운영 가능.
- **일반 노드:** 사용자의 Stellavault 인스턴스. 접속과 종료를 반복. 온라인 시 임베딩 공유.
- **프리미엄 릴레이:** 오프라인 노드를 위해 임베딩 인덱스를 캐시하는 선택적 상시 가동 노드($5/월 서비스).

#### 20.3 연결 흐름

```
1. Node A가 연합이 활성화된 상태로 Stellavault 시작
2. Node A가 DHT에 공지: "나는 federation-group:<topic-hash>의 일원입니다"
3. DHT가 같은 토픽을 공지한 피어들을 반환
4. Node A가 Noise 암호화 TCP를 통해 피어에 연결 (홀 펀칭)
5. 핸드셰이크: 노드 신원(Ed25519 공개키), 기능, 임베딩 모델 정보 교환
6. 상호 인증: 서명 확인, 신뢰 점수 확인
7. 쿼리 준비 완료
```

### 21. 임베딩 공유 프로토콜

#### 21.1 공유되는 것 (그리고 공유되지 않는 것)

| 공유됨 | 공유되지 않음 |
|--------|--------------|
| 임베딩 벡터 (384차원 float32, DP 섭동 적용) | 원시 텍스트 콘텐츠 |
| 문서 메타데이터: 제목, 토픽 태그, 단어 수 | 전체 문서 본문 |
| FSRS 회수 가능성 점수 (집계) | 개인 주석, 하이라이트 |
| 지식 갭 영역 (클러스터 중심) | 파일 경로, 폴더 구조 |
| 마지막 업데이트 타임스탬프 | 편집 이력 |
| 노드 기능 선언 | 사용자 신원 상세 |

#### 21.2 임베딩 동기화 프로토콜

```protobuf
// stellavault-federation.proto

syntax = "proto3";

package stellavault.federation;

// 노드가 임베딩 카탈로그를 공지
message EmbeddingCatalogue {
  string node_id = 1;           // Ed25519 공개키 (hex)
  uint64 version = 2;           // 단조 증가 버전 카운터
  uint32 embedding_dim = 3;     // 예: all-MiniLM-L6-v2의 경우 384
  string embedding_model = 4;   // 호환성을 위한 모델 식별자
  uint32 document_count = 5;    // 볼트의 총 문서 수
  repeated TopicSummary topics = 6;
  bytes signature = 7;          // 필드 1-6에 대한 Ed25519 서명
}

message TopicSummary {
  string topic = 1;             // 토픽 태그
  uint32 count = 2;             // 이 토픽의 문서 수
  repeated float centroid = 3;  // 클러스터 중심 (DP 섭동 적용)
  float avg_retrievability = 4; // 이 토픽의 FSRS 평균
}

// 연합 검색 쿼리
message SearchQuery {
  string query_id = 1;          // UUID
  string requester_id = 2;      // 노드 ID
  repeated float query_embedding = 3;  // 쿼리 벡터 (384차원)
  uint32 limit = 4;             // 노드당 최대 결과 수
  float min_similarity = 5;     // 임계값 (기본값 0.5)
  repeated string topic_filter = 6;    // 선택적 토픽 필터
  uint32 hop_limit = 7;         // 최대 전달 홉 수 (기본값 2)
  bytes signature = 8;
}

// 원격 노드의 검색 결과
message SearchResult {
  string query_id = 1;
  string responder_id = 2;
  repeated ScoredDocument documents = 3;
  uint32 total_searched = 4;    // 검색된 문서 수
  uint64 latency_ms = 5;
  bytes signature = 6;
}

message ScoredDocument {
  string document_id = 1;       // 불투명 ID (파일 경로 아님)
  string title = 2;
  repeated string topics = 3;
  float similarity_score = 4;
  float retrievability = 5;     // FSRS 점수
  uint32 word_count = 6;
  string last_updated = 7;      // ISO 8601
  // 참고: 콘텐츠 필드 없음. 절대 전송되지 않음.
}

// 지식 라우팅
message RouteQuery {
  string query_id = 1;
  repeated float query_embedding = 2;
  uint32 max_routes = 3;
}

message RouteResult {
  string query_id = 1;
  repeated NodeRoute routes = 2;
}

message NodeRoute {
  string node_id = 1;
  float relevance_score = 2;    // 토픽 중심 유사도 기반
  repeated string relevant_topics = 3;

  bool online = 5;
}

// 신뢰 증명
message TrustAttestation {
  string attester_id = 1;       // 보증하는 자
  string attestee_id = 2;       // 보증받는 자
  float trust_level = 3;        // 0.0 ~ 1.0
  string reason = 4;            // 선택 사항
  string timestamp = 5;
  bytes signature = 6;
}
```

#### 21.3 동기화 전략

**초기 동기화:**
1. 피어와 첫 연결 시, `EmbeddingCatalogue` 메시지 교환
2. 각 노드가 피어 카탈로그를 로컬 `federation.db`(별도 SQLite)에 저장
3. 대량 임베딩 전송 없음 -- 카탈로그(토픽 요약 + 중심)만

**쿼리 시점 가져오기:**
- 연합 검색 쿼리가 도착하면, 응답 노드가 자신의 로컬 임베딩에 대해 유사도 계산
- 상위 K개 결과(메타데이터만) 반환
- 노드 간 전체 임베딩 셋 전송을 방지

**증분 업데이트:**
- 볼트가 변경되면 노드가 `CatalogueUpdate` 메시지를 브로드캐스트(디바운싱, 최대 1시간에 1회)
- 업데이트는 전체 카탈로그가 아닌 변경된 토픽 요약만 포함
- 버전 카운터가 순서를 보장

**오프라인 처리:**
- 각 노드가 피어 카탈로그를 로컬에 캐시
- 피어가 오프라인일 때도 캐시된 카탈로그를 지식 라우팅에 사용
- 프리미엄 릴레이가 구독자를 위해 항상 최신 카탈로그를 유지

### 22. 프라이버시 레이어

#### 22.1 위협 모델

| 위협 | 심각도 | 완화 |
|------|:------:|------|
| **임베딩 역변환 공격** | 심각 | 차등 프라이버시 노이즈 + EGuard 투영 + 소형 모델 (384차원은 1024/1536차원보다 역변환이 어려움) |
| **멤버십 추론** | 높음 | 노이즈 주입으로 특정 문서에 대한 합리적 부인 가능성 보장 |
| **토픽 핑거프린팅** | 중간 | 토픽 중심은 집계됨(문서별이 아님), DP 노이즈 적용 |
| **트래픽 분석** | 중간 | 커버 트래픽(일정 간격 랜덤 쿼리), 민감한 쿼리에 양파 라우팅 |
| **악의적 노드** | 높음 | 신뢰 시스템, 쿼리 속도 제한, 이상 탐지 |
| **네트워크 메타데이터** | 낮음 | Hyperswarm Noise 암호화, 중앙 서버 로깅 없음 |

#### 22.2 차등 프라이버시 구현

```
프라이버시 예산: epsilon = 1.0 (노드별 설정 가능)

각 공유 임베딩 벡터 e에 대해:
1. 민감도 계산: delta_f = 문서 하나 추가/제거 시 최대 L2 변화
2. 가우시안 노이즈 생성: n ~ N(0, sigma^2 * I)
   여기서 sigma = delta_f * sqrt(2 * ln(1.25 / delta)) / epsilon
3. 섭동된 임베딩: e' = e + n
4. 정규화: e'' = e' / ||e'||

토픽 중심의 경우:
- 개별 임베딩이 아닌 클러스터 중심에 DP 적용
- 중심은 덜 민감하므로 더 높은 노이즈 예산(epsilon = 2.0)

사용자 설정 가능 프라이버시 수준:
- "paranoid"  : epsilon = 0.1 (매우 노이즈 많음, 낮은 검색 품질)
- "balanced"  : epsilon = 1.0 (기본값, 프라이버시/품질 균형)
- "open"      : epsilon = 10.0 (최소 노이즈, 최상 검색 품질)
- "public"    : DP 없음 (의도적으로 공개하는 지식 볼트용)
```

#### 22.3 EGuard 투영 (선택적, 고급)

더 강력한 보장이 필요한 사용자를 위해:
- 임베딩을 프라이버시 보존 부분공간으로 매핑하는 트랜스포머 기반 투영 네트워크 훈련
- 이 투영은 역변환 공격의 95% 이상을 차단하면서 검색 정확도 98% 이상 유지
- 연합 패키지에 사전 훈련된 모델로 포함
- 쿼리당 ~50ms 지연 추가

#### 22.4 임베딩 모델 고려사항

현재 기본 모델 `all-MiniLM-L6-v2` (384차원)은 자연적인 프라이버시 이점을 가집니다:
- 더 작은 차원 = 더 적은 정보 = 역변환이 더 어려움
- 2024년 연구에 따르면 대형 모델의 32토큰 입력에서 92% 복원이 가능하지만, 384차원 모델은 유의미하게 낮은 복원율을 보임
- v1에서 이 모델로 표준화하면 교차 모델 정렬 복잡성을 회피
- 향후: 검증된 정렬 행렬이 있는 모델 레지스트리 지원

### 23. 검색 라우팅

#### 23.1 "이 질문은 노드 C가 가장 잘 답합니다"의 작동 원리

```
쿼리 라우팅 알고리즘:

1. Node A의 사용자가 질문: "CRISPR 프라이머 설계 모범 사례는?"

2. Node A가 쿼리 임베딩 계산: q = embed("CRISPR primer design...")

3. Node A가 알려진 모든 노드의 로컬 카탈로그 캐시 확인:
   연합 내 각 노드 N에 대해:
     N.catalogue의 각 토픽 T에 대해:
       score(N, T) = cosine(q, T.centroid) * T.count * N.trust_score * uptime_factor(N)

4. 최대 토픽 점수로 노드 순위 지정:
   Node C: 0.89 (토픽: "molecular-biology", 340개 문서, 신뢰: 0.95)
   Node B: 0.72 (토픽: "genetics", 120개 문서, 신뢰: 0.88)
   Node E: 0.45 (토픽: "lab-protocols", 80개 문서, 신뢰: 0.91)

5. 상위 K개 노드에 SearchQuery 전송 (기본 K=5, 설정 가능)

6. RRF(로컬 하이브리드 검색과 동일한 알고리즘)를 사용하여 결과 병합:
   모든 노드의 각 결과에 대해:
     rrf_score = sum(1 / (k + rank_in_node)) * node_trust_weight

7. 병합된 결과를 사용자/AI 에이전트에 반환
```

#### 23.2 멀티홉 라우팅

대규모 네트워크(50+ 노드)의 경우:
- 직접 연결이 제한됨(각 노드는 최대 ~20개 피어에 연결)
- 쿼리에 `hop_limit` 포함(기본값 2)
- 각 릴레이 노드가 자신의 피어에 쿼리를 전달할 수 있음
- 결과가 같은 경로로 돌아옴
- `query_id` 중복 제거를 통한 루프 탐지

### 24. 신뢰 모델

#### 24.1 Web of Trust

```
신뢰 점수 계산:

trust(A, B) = direct_trust(A, B) * 0.6 + transitive_trust(A, B) * 0.3 + karma(B) * 0.1

여기서:
- direct_trust: A에서 B로의 명시적 보증 (0.0 - 1.0)
- transitive_trust: 모든 C에 대해 max(trust(A, C) * trust(C, B))
  (신뢰가 홉을 통해 곱셈적으로 감쇠, 최대 3홉)
- karma: 다음에 기반한 글로벌 평판:
  * 가동 시간 비율 (30%)
  * 검색 결과 품질 피드백 (40%)
  * 노드 나이 (10%)
  * 보증 수 (20%)
```

#### 24.2 신뢰 연산

| 연산 | 설명 | CLI 명령 |
|------|------|----------|
| **보증** | 노드를 명시적으로 신뢰 | `sv federate vouch <node-id> --level 0.8` |
| **철회** | 신뢰 증명 제거 | `sv federate revoke <node-id>` |
| **차단** | 노드 차단 (신뢰 0, 쿼리 불가) | `sv federate block <node-id>` |
| **검사** | 노드의 신뢰 체인 확인 | `sv federate trust <node-id>` |

#### 24.3 Sybil 저항

- **지식 증명:** 새 노드는 100개 이상의 문서와 30일 이상의 색인 이력이 있는 볼트가 있어야 연합에 참여 가능
- **보증 요구:** 신뢰 >0.5인 기존 노드로부터 최소 1회 보증
- **속도 제한:** 새 노드는 처음 7일간 시간당 10회 쿼리로 제한
- **이상 탐지:** 갑작스러운 대량 문서 추가 시 검토 플래그 트리거

```sql
  document_count INTEGER DEFAULT 0,
  trust_score REAL DEFAULT 0.0,
  is_blocked INTEGER DEFAULT 0,
  joined_at TEXT NOT NULL
);

-- 연합 그룹
CREATE TABLE groups (
  group_id TEXT PRIMARY KEY,         -- 그룹명 + 생성자의 SHA256
  name TEXT NOT NULL,
  description TEXT,
  created_by TEXT NOT NULL,
  joined_at TEXT NOT NULL,
  invite_code TEXT UNIQUE
);

-- 그룹 멤버십
CREATE TABLE group_members (
  group_id TEXT NOT NULL,
  node_id TEXT NOT NULL,
  role TEXT DEFAULT 'member',        -- 'admin', 'member'
  joined_at TEXT NOT NULL,
  PRIMARY KEY (group_id, node_id)
);

-- 신뢰 증명
CREATE TABLE trust_attestations (
  attester_id TEXT NOT NULL,
  attestee_id TEXT NOT NULL,
  trust_level REAL NOT NULL,
  reason TEXT,
  created_at TEXT NOT NULL,
  signature BLOB NOT NULL,
  PRIMARY KEY (attester_id, attestee_id)
);

-- 캐시된 피어 카탈로그 (피어 오프라인 시 라우팅용)
CREATE TABLE peer_catalogues (
  node_id TEXT NOT NULL,
  topic TEXT NOT NULL,
  document_count INTEGER,
  centroid BLOB,                     -- Float32 배열 (DP 섭동 적용)
  avg_retrievability REAL,
  updated_at TEXT NOT NULL,
  PRIMARY KEY (node_id, topic)
);

-- 검색 크레딧
CREATE TABLE credits (
  node_id TEXT PRIMARY KEY,
  balance INTEGER DEFAULT 100,
  total_earned INTEGER DEFAULT 0,
  total_spent INTEGER DEFAULT 0
);

-- 쿼리 로그 (로컬 전용, 분석용)
CREATE TABLE query_log (
  query_id TEXT PRIMARY KEY,
  query_text TEXT,
  nodes_queried INTEGER,
  results_returned INTEGER,
  latency_ms INTEGER,
  timestamp TEXT NOT NULL
);
```

## 부록

### A. 기술 결정 요약

| 결정 | 선택 | 근거 |
|------|------|------|
| P2P 프로토콜 | Hyperswarm | 최상의 NAT 처리, Node.js 네이티브, 가장 간단한 통합 |
| 직렬화 | Protobuf | 임베딩 벡터에 효율적(384 float32 = 1.5KB), 스키마 진화 지원 |
| 신원 | Ed25519 + 선택적 DID:key | 빠른 서명, 작은 키, 향후 상호운용을 위한 DID 호환 |
| 프라이버시 | 차등 프라이버시(가우시안) + 선택적 EGuard | 검증된 이론, 조절 가능, 고급 사용자를 위한 EGuard |
| 신뢰 | Web of Trust (보증 + 카르마) | 탈중앙화, 중앙 권한 없음, 유기적 성장 |
| 임베딩 모델 (v1) | all-MiniLM-L6-v2 (384차원) | 이미 사용 중, 작은 차원 = 더 나은 프라이버시, 충분한 품질 |
| 저장소 | 별도 federation.db (SQLite) | 메인 index.db와 격리, 독립적 생명주기 |

### C. 용어집

| 용어 | 정의 |
|------|------|
| **Federation** | 독립성을 유지하면서 협력하는 Stellavault 노드 네트워크 |
| **Node** | 연합을 실행하는 단일 Stellavault 인스턴스 |
| **Embedding Catalogue** | 노드 지식의 요약(토픽 중심, 수, FSRS 점수) |
| **DP (Differential Privacy)** | 통계적 유용성을 보존하면서 데이터에 노이즈를 추가하는 수학적 프레임워크 |
| **EGuard** | 임베딩 역변환 공격을 차단하는 트랜스포머 기반 투영 네트워크 |
| **Hyperswarm** | 내장 DHT와 NAT 홀 펀칭을 갖춘 P2P 네트워킹 라이브러리 |
| **Noise Protocol** | 보안 채널을 위한 암호화 핸드셰이크 프레임워크 |
